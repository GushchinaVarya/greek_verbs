# greek_database_bot — логика и знания о проекте

Этот документ фиксирует **полную логику работы проекта** (пользовательский бот тренировки + админ-бот управления базой), структуру данных и правила эксплуатации.

## 1) Что делает проект (кратко)

Проект предназначен для тренировки спряжения/форм греческих глаголов.

- **Пользовательский бот** (`trainbot.py`):  
  - показывает меню действий,
  - позволяет **прочитать правила** (присылает файлы с правилами по выбранному времени/наклонению),
  - позволяет **тренироваться**: присылает случайную русскую фразу, даёт подсказку и ответ (в спойлерах), кнопки «ещё / завершить».

- **Админ-бот** (`adminbot.py`):  
  - выгружает таблицы заданий в CSV,
  - добавляет/обновляет записи в базе (через ввод строки в заданном формате),
  - предупреждает о дубликатах по полю `question`.

Данные хранятся в SQLite:

- `greek_db_all.db` — база заданий (`present/future/past/imperative`).
- `users_all.db` — список пользователей (для регистрации и рассылок).

## 2) Структура репозитория

Ключевые файлы:

- `trainbot.py` — входная точка **пользовательского** Telegram-бота (polling).
- `adminbot.py` — входная точка **админского** Telegram-бота (polling).
- `db_functions.py` — общий слой работы с SQLite: выбор вопроса, генерация CSV, вставка/обновление строк, учёт пользователей.
- `greek_db_all.db` — SQLite база с заданиями.
- `users_all.db` — SQLite база пользователей.
- `pip-requirements.txt` — зависимости.
- `create_db.py` — “черновик” для ручных операций с БД (создание/миграции/отладка запросов; много закомментировано).
- `add_to_db_with_csv.py` — импорт записей в SQLite из CSV (CLI-скрипт).
- `add_to_db_with_str.py` — пример массовой вставки из строк (CLI-скрипт).

Важное про конфигурацию:

- `config.py` используется обоими ботами, но **исключён из git** (`.gitignore`).  
  Это означает, что в репозитории по умолчанию нет токенов/локальных путей и проект без `config.py` не запустится.

## 3) Зависимости и окружение

`pip-requirements.txt`:

- `python-telegram-bot==22.2` — Telegram Bot API (ConversationHandler, inline-кнопки).
- `pandas` — генерация CSV.

Фактически в коде также используется:

- `numpy` (в `db_functions.py` для случайного выбора `rowid`).

Рекомендуется держать `numpy` установленным (или добавить в зависимости).

## 4) Конфигурация (`config.py`)

Файл `config.py` не в репозитории, но по импорту видно, какие значения должны быть заданы.

### 4.1) Переменные для `trainbot.py`

- `TG_TOKEN_TRAIN`: токен Telegram-бота тренировки.
- `PRESENT_RULES_FILES`, `PAST_RULES_FILES`, `FUTURE_RULES_FILES`, `IMPERATIVE_RULES`:  
  **списки путей** к файлам (обычно PDF/документы), которые бот отправляет как “правила”.
- `ADMINS_IDS`: список `chat_id` админов (int) для админ-команды рассылки.
- `UPDATE_TEXT`: текст рассылки (используется в `/announcement`).

### 4.2) Переменные для `adminbot.py`

- `TG_TOKEN_ADMIN`: токен Telegram-бота администрирования.

### 4.3) Переменные для `db_functions.py`

- `DB_NAME`: имя/путь к базе заданий (обычно `greek_db_all.db`).
- `DB_USERS_NAME`: имя/путь к базе пользователей (обычно `users_all.db`).

Важно:

- **Не храните токены в git**. В идеале: `.env` + загрузка из переменных окружения.

## 5) Данные и схема БД

### 5.1) `greek_db_all.db` — база заданий

Таблицы:

- `present` — настоящее время
- `future` — будущее время
- `past` — прошедшее время
- `imperative` — повелительное наклонение

Колонки (в каждой таблице):

- `question` (TEXT, NOT NULL): русская фраза-задание (то, что переводят).
- `answer` (TEXT, NOT NULL): греческий перевод/форма.
- `comment` (TEXT): пояснение (например тип глагола/правило/род и т.д.).
- `hint` (TEXT, DEFAULT ...): подсказка (обычно “используйте глагол ...”).

Смысл хранения:

- Каждая строка — отдельная карточка тренировки.
- Выборка для тренировки идёт случайно по `rowid`.

### 5.2) `users_all.db` — база пользователей

Таблица:

- `users(chat_id BIGINT PRIMARY KEY, name TEXT)`

Назначение:

- регистрировать пользователей при `/start` (чтобы знать, кому рассылать обновления),
- получать список `chat_id` для рассылки в `/announcement`.

## 6) Общая логика работы с таблицами (маппинг “русское → имя таблицы”)

`db_functions.get_tablename(table: str)` сопоставляет текст выбора (на русском) с реальным именем таблицы:

- `"настоящее время"` → `present`
- `"будущее время"` → `future`
- `"прошедшее время"` → `past`
- `"повелительное наклонение"` → `imperative`

Это важно: **в UI ботов везде фигурируют русские строки**, а фактические SQL-запросы идут в английские имена таблиц.

## 7) Пользовательский бот (`trainbot.py`) — полный флоу

### 7.1) Общая идея интерфейса

Бот построен на `ConversationHandler` и inline-кнопках. Состояния диалога сохраняются через `PicklePersistence` в файл `bot_persistence.pkl` — при перезапуске бота пользователи не теряют свой прогресс и могут продолжать нажимать кнопки.

**Приветственный текст** (`/start`, первый запуск):
> Привет! Я помогу тренировать греческие глаголы.
> Выберите: Правила — чтобы вспомнить спряжения, или Тренировка — чтобы практиковаться.
> Остановить тренировку: /stop

**Текст меню:**
> Что вы хотите сделать?
> Выберите режим: правила или тренировка

Верхний уровень:

- `/start` → меню:
  - «📘 Правила»
  - «🧠 Тренировка»

Дальше — выбор времени/наклонения:

- «Настоящее время»
- «Повелительное наклонение»
- «Будущее время»
- «Прошедшее время»

Зависит от ветки:

- **📘 Правила** → отправка файлов правил → возврат в меню
- **🧠 Тренировка** → инструктаж → «▶️ Начать тренировку» → выдача карточек → «Еще / Завершить»

### 7.2) Регистрация пользователя

При первом `/start` бот вызывает `add_user(chat_id, first_name)` и сохраняет пользователя в `users_all.db`.  
Это позволяет потом делать рассылку через `/announcement`.

### 7.3) Ветка "📘 Правила"

1) Пользователь нажимает «📘 Правила».
2) Бот просит выбрать время/наклонение.
3) По выбору определяет список файлов (из `config.py`) и отправляет их как документы в чат.
4) После отправки предлагает дальнейшее действие (снова кнопки “правила / тренировка”).

Особенности:

- В коде выбор файлов сделан через сравнение текста таблицы с русскими строками.
- Отправка файлов выполняется в цикле `for filename in files: send_document(...)`.

### 7.4) Ветка "🧠 Тренировка"

1) Пользователь нажимает «🧠 Тренировка».
2) Выбирает время/наклонение.
3) Бот присылает инструкцию (MarkdownV2, жирный заголовок + нумерованные пункты с эмодзи) и кнопки:
   - «▶️ Начать тренировку» (получить первый вопрос)
   - «↩️ Назад»

Текст инструкции:

> **{table}**
>
> Как тренироваться:
>
> 1. 🗒 Приготовьте ручку и бумагу
> 2. Я пришлю фразу по-русски — переведите на греческий
> 3. Не знаете глагол? 💡 Откройте подсказку
> 4. Откройте ответ ✅ и проверьте себя
>
> _Не присылайте ответ боту — вы сами лучший судья._
> _Не забывайте про ударения!_

4) На каждую выдачу вопроса бот отправляет **три сообщения**:
   - **вопрос**: “переведите на греческий” + `question`
   - **подсказка**: `hint` в spoiler-формате `||...||`
   - **ответ**: `answer` и `comment` в spoiler-формате `||...||` + кнопки:
     - «Еще» (новая карточка)
     - «Завершить» (закончить тренировку)

### 7.5) Команды

- `/start` — старт/перезапуск сценария и регистрация пользователя.
- `/stop` — завершение тренировки (выход из диалога).

### 7.6) Админ-команда рассылки

`/announcement`:

- доступна только если `chat_id` отправителя есть в `ADMINS_IDS` (иначе — тихо игнорируется, в лог пишется `WARNING`),
- берёт всех пользователей через `get_users()` из `users_all.db`,
- **перед рассылкой** отправляет `UPDATE_TEXT` самому администратору как тест на корректность MarkdownV2:
  - если текст корректен — рассылает всем с `parse_mode='MarkdownV2'`,
  - если возникает `BadRequest` (неэкранированные спецсимволы) — логирует предупреждение с причиной и рассылает всем как plain text,
- затем просит каждого пользователя нажать `/start` для перезапуска,
- в лог выводится итоговая сводка: сколько доставлено, заблокировано, упало с ошибкой.

**Важно про `UPDATE_TEXT`:** если нужно форматирование (жирный, курсив), текст должен быть написан по правилам MarkdownV2 — спецсимволы `. - ! ( ) [ ] _ * ~ > # + = | { }` должны быть экранированы обратным слешем.

## 8) Админ-бот (`adminbot.py`) — полный флоу

### 8.1) Варианты действий

`/start` → меню:

- «Скачать таблицу глаголов»
- «Дополнить таблицу глаголов»

Дальше в обоих случаях выбирается таблица (время/наклонение).

### 8.2) Выгрузка таблицы в CSV

1) Пользователь выбирает «Скачать таблицу глаголов».
2) Выбирает таблицу (present/future/past/imperative в русских строках).
3) `generate_csv(table)`:
   - читает все строки из нужной таблицы,
   - формирует DataFrame с колонками `question, answer, comment, hint`,
   - пишет файл в `tmp_db/<tablename>_<timestamp>.csv`,
   - возвращает имя файла.
4) Бот отправляет CSV документом в чат.

Технические требования:

- директория `tmp_db/` должна существовать (она в `.gitignore`, но должна быть на диске).

### 8.3) Добавление/обновление строки

1) Пользователь выбирает «Дополнить таблицу глаголов».
2) Выбирает таблицу.
3) Бот просит ввести строку в формате:

`глагол по русски, глагол по гречески, комментарий, подсказка`

Пример (как задумывается по тексту бота):

- вопрос: “он покупает”
- ответ: “αυτός αγοράζει”
- комментарий: “правильный глагол мужской род”
- подсказка: “используйте глагол αγοράζω”

4) Бот парсит текст `split(',')` и сохраняет 4 поля в `context.user_data`.

Важное:

- Формат **жёсткий**: требуется ровно 4 части, разделённых запятыми.
- Текст приводится к `strip().lower()`.

5) Проверка дубликата:

`check_if_db_has_this(table, question)` ищет строки по `question = ?`:

- если уже есть — возвращает текст “в базе уже есть …” и включает опцию “удалить старую запись”.

6) Запись:

`write_to_table(table, question, answer, comment, hint)`:

- ищет строки с таким `question`,
- если есть — удаляет старые,
- вставляет новую строку.

Особенность реализации:

- SQL собирается через f-string с подстановкой строк. Это работает для “обычных” данных, но в общем случае несёт риск SQL-инъекций/ошибок экранирования при наличии кавычек в тексте.

## 9) `db_functions.py` — знания о внутренних функциях

### 9.1) `get_question(table: str) -> (text_q, text_h, text_a)`

Логика:

- берёт все `rowid` из таблицы,
- случайно выбирает один через `numpy.random.choice`,
- читает `question, answer, comment, hint`,
- формирует три текста:
  - вопрос (с MarkdownV2),
  - подсказка (spoiler),
  - ответ + комментарий (spoiler).

### 9.2) `generate_csv(table: str) -> (filename, ready_flag)`

- выгружает все строки таблицы,
- пишет CSV в `tmp_db/`,
- возвращает имя файла.

### 9.3) `add_user(chat_id, name) -> bool`

- проверяет наличие `chat_id` в `users`,
- если нет — вставляет и возвращает `True` (новый пользователь).

### 9.4) `get_users() -> (ids, names)`

- возвращает список `chat_id` и имён.

## 11) Логгирование (`trainbot.py`)

### 11.1) Конфигурация

Логи пишутся **одновременно в два места**:
- файл `log_trainig.log` (в корне проекта, кодировка UTF-8),
- консоль (stdout).

Формат строки:
```
2026-07-09 15:52:01 | INFO     | [QUESTION] Варвара (id=123274089) requested next card — table='настоящее время'
```

Библиотеки `telegram` и `httpx` подавлены до уровня `WARNING`, чтобы не засорять лог техническими деталями соединений.

### 11.2) Теги и что они означают

Каждое сообщение в логе начинается с тега — по нему удобно фильтровать:

| Тег | Когда пишется |
|---|---|
| `[START]` | `/start` — новый или вернувшийся пользователь |
| `[MENU]` | Нажата кнопка «Правила» или «Тренировка» |
| `[TRAIN]` | Выбрана таблица для тренировки |
| `[QUESTION]` | Запрошена новая карточка |
| `[RULES]` | Запрошены файлы с правилами (каждый файл логируется отдельно) |
| `[FINISH]` | Нажата кнопка «Завершить» |
| `[STOP]` | Отправлена команда `/stop` |
| `[ANNOUNCEMENT]` | Любое действие в рамках рассылки |

### 11.3) Уровни логов

- `INFO` — нормальный ход работы (пользователь нажал кнопку, карточка отправлена и т.д.)
- `WARNING` — нештатная, но не критичная ситуация (unauthorized `/announcement`, пользователь заблокировал бота, MarkdownV2 некорректен)
- `ERROR` / `EXCEPTION` — ошибка с полным traceback (не удалось получить вопрос из БД, не удалось отправить сообщение, не удалось отправить файл с правилами)

### 11.4) Вспомогательная функция `_u(update)`

Все логи используют хелпер `_u(update)`, который возвращает строку вида `Варвара Соологуб (id=123274089)`. Это позволяет отслеживать все действия конкретного пользователя через поиск по `id=`.

## 12) Негласные требования/подводные камни

- **`bot_persistence.pkl`**: файл создаётся автоматически при первом запуске `trainbot.py` и хранит состояния всех диалогов (`ConversationHandler`) и `user_data`. Не добавляйте его в git (уже в `.gitignore`). При необходимости сбросить все состояния — удалите файл перед запуском.
- **`tmp_db/`**: должна существовать, иначе выгрузка CSV упадёт.
- **`config.py`**: должен быть создан локально (он в `.gitignore`).
- **База данных**: должна быть рядом/доступна по пути `DB_NAME`/`DB_USERS_NAME`.
- **MarkdownV2**: сообщения в тренботе и админботе иногда отправляются с `parse_mode='MarkdownV2'`; если в данных встречаются спецсимволы, может ломаться форматирование.
- **SQL через f-string**: в `write_to_table()` и части других функций строки подставляются напрямую — кавычки в данных могут ломать запрос; потенциально небезопасно.

## 12) Что считать “знаниями” бота (семантика данных)

Каждая запись в таблице означает:

- **question**: что пользователь должен перевести/просклонять.
- **hint**: “опорный” глагол/форма, на основе которой пользователь должен построить ответ (подсказка по правилу).
- **answer**: эталонный ответ на греческом.
- **comment**: краткое объяснение (тип глагола, правило, особенности).

Цель тренировки — не автоматическая проверка, а **самопроверка**: пользователь пишет на бумаге и открывает ответ/подсказку по необходимости.
